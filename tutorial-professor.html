Making Pygame Games Accessible

1. Course Overview & Learning Objectives

This module teaches students to build a simple arcade game in Pygame, then critically evaluate and improve its accessibility. Students learn that accessibility is not an afterthought — it's a design principle that makes software better for all users.

Learning Objectives

By the end of this module, students will be able to:

  • Explain the game loop pattern and implement it in Pygame
  • Use object-oriented programming to create Player and Enemy sprites
  • Identify accessibility barriers in software using WCAG AA criteria
  • Calculate and evaluate color contrast ratios for visual accessibility
  • Implement concrete accessibility features: adjustable speed, readable fonts, volume control, and high-contrast visuals
  • Articulate why each accessibility change matters and who it helps

2. Prerequisites & Setup

Student Prerequisites

  • Basic Python knowledge: variables, functions, loops, conditionals
  • Familiarity with classes is helpful but not required (Lesson 2 introduces OOP)
  • No prior Pygame or game development experience needed

Environment Setup

Install Pygame
pip install pygame

Required Files

Students will need image assets (Player.png, Enemy.png, background image) and a crash sound file (crash.wav). These can be provided as a starter zip, or students can create simple colored rectangles programmatically.

Instructor Tip

Consider distributing a "starter kit" zip file with the base (inaccessible) game already working. This lets Lesson 3 focus entirely on the audit and Lesson 4 on the fixes, rather than debugging setup issues.

3. Lesson 1: Pygame Foundations & the Game Loop

Suggested Timeline (50 min)

0–10 min
Warm-up: Ask students to name their favorite game. What makes it enjoyable? What would happen if they couldn't see the colors, or the game moved too fast?
10–20 min
Lecture: Introduce Pygame — importing, initializing, the display surface
20–35 min
Live code: Build the game loop, explain FPS, demonstrate coordinate system
35–45 min
Hands-on: Students create a window, set a background color, and display a rectangle that moves
45–50 min
Wrap-up: Preview next session's OOP approach

Key Concepts to Cover

Initializing Pygame:

import pygame, sys
from pygame.locals import *

pygame.init()

Explain that pygame.init() starts the engine and must come before any Pygame code. The pygame.locals import gives access to constants like QUIT, K_LEFT, etc.

The Game Loop:

#Game loop begins
while True:
    for event in pygame.event.get():
        if event.type == QUIT:
            pygame.quit()
            sys.exit()

    # Game logic goes here

    pygame.display.update()
    FramePerSec.tick(60)
Discussion Prompt

"What would happen if we didn't limit the frame rate? Why might different players need different speeds?" This seeds the accessibility conversation for Lesson 3.

Colors & the Coordinate System:

# RGB color tuples (0-255 per channel)
BLACK = (0, 0, 0)
WHITE = (255, 255, 255)
RED   = (255, 0, 0)

# Screen: (0,0) is top-left, x grows right, y grows down
DISPLAYSURF = pygame.display.set_mode((400, 600))

Frames Per Second:

# FPS controls game speed — 60 is standard, 30 is "slow mode"
normal_mode = 60
slow_mode = 30
FPS = normal_mode
FramePerSec = pygame.time.Clock()

Explain that movies run at 24 FPS. Games typically aim for 30–60 FPS. Without a limiter, the loop runs as fast as the CPU allows, causing inconsistent behavior.

4. Lesson 2: Building the Game — Sprites, Classes & Collision

Suggested Timeline (50 min)

0–10 min
Review: Quick recap of the game loop. Show the starting code from Lesson 1.
10–25 min
Lecture + live code: Introduce OOP through the Player and Enemy classes
25–40 min
Hands-on: Students implement sprite groups, collision detection, and scoring
40–50 min
Demo & discussion: Play the completed (inaccessible) game. Ask: "What could make this hard for some players?"

The Player Class

class Player(pygame.sprite.Sprite):
    def __init__(self):
        super().__init__()
        self.image = pygame.image.load("Player.png")
        self.rect = self.image.get_rect()
        self.rect.center = (160, 520)

    def move(self):
        pressed_keys = pygame.key.get_pressed()
        if self.rect.left > 0:
            if pressed_keys[K_LEFT]:
                self.rect.move_ip(-5, 0)
        if self.rect.right < SCREEN_WIDTH:
            if pressed_keys[K_RIGHT]:
                self.rect.move_ip(5, 0)
Teaching Note: OOP for Beginners

If students haven't seen classes before, frame it as a "blueprint." The Player class is a blueprint for creating a player object. self refers to "this specific player." __init__ is what happens when a new player is created. Methods like move() are actions the player can take.

The Enemy Class

class Enemy(pygame.sprite.Sprite):
    def __init__(self):
        super().__init__()
        self.image = pygame.image.load("Enemy.png")
        self.rect = self.image.get_rect()
        self.rect.center = (random.randint(40, SCREEN_WIDTH-40), 0)

    def move(self):
        global SCORE
        self.rect.move_ip(0, SPEED)
        if self.rect.top > 600:
            SCORE += 1
            self.rect.top = 0
            self.rect.center = (random.randint(40, SCREEN_WIDTH - 40), 0)

Sprite Groups & Collision Detection

# Groups let us manage many sprites at once
enemies = pygame.sprite.Group()
enemies.add(E1)
all_sprites = pygame.sprite.Group()
all_sprites.add(P1)
all_sprites.add(E1)

# In the game loop — check if player hit any enemy
if pygame.sprite.spritecollideany(P1, enemies):
    # Game over!

Explain that spritecollideany() checks if the first sprite's rect overlaps with any sprite in the group. This scales to hundreds of enemies without changing the code.

Custom Events & Increasing Difficulty

INC_SPEED = pygame.USEREVENT + 1
pygame.time.set_timer(INC_SPEED, 1000)

# In the game loop:
if event.type == INC_SPEED:
    SPEED += 0.5
End-of-Lesson Discussion

Have students play the finished (inaccessible) version. Ask them to identify problems:

  • "Can you read the score easily?"
  • "What if someone is color blind — can they tell the player from the background?"
  • "What if the game is too fast for someone to react?"
  • "What if the crash sound is too loud or too quiet?"

Record their observations on the board. These become the "fix list" for Lessons 3–4.

5. Lesson 3: Accessibility Audit — Applying WCAG to Games

Suggested Timeline (50 min)

0–15 min
Lecture: Introduction to WCAG AA. What is accessibility? Who benefits?
15–30 min
Activity: Students audit the game using the WCAG checklist below
30–40 min
Group work: Teams present findings and propose solutions
40–50 min
Planning: Each student creates a prioritized fix list for Lesson 4

Introduction to WCAG for Games

The Web Content Accessibility Guidelines (WCAG) are the global standard for digital accessibility. While written for web content, the principles apply directly to games:

WCAG PrincipleGame Application
PerceivableCan players see/hear all important game elements? Are colors distinguishable?
OperableCan players control the game effectively? Is the speed manageable?
UnderstandableIs text readable? Are game states clear (score, game over)?
RobustDoes the game work with different input methods and configurations?

Contrast: The Most Common Failure

WCAG AA requires:

  • Text contrast: at least 4.5:1 against the background
  • Non-text elements (game objects, UI): at least 3:1
Hands-On Activity: Contrast Audit

Have students use WebAIM's Contrast Checker to evaluate the original game's colors:

  • Grey road (#808080) vs. white lines (#FFFFFF) → 1.86:1 (FAIL)
  • Grey road vs. yellow lines (#FFFF00) → 1.49:1 (FAIL)
  • Grey road vs. blue car (#0000FF) → 1.07:1 (FAIL)

Then have them find accessible color combinations that pass 3:1 for objects and 4.5:1 for text.

Full Accessibility Checklist for Student Audit

CategoryQuestionWCAG Criterion
ContrastDo all game objects meet 3:1 contrast against the background?1.4.11
ContrastDoes all text meet 4.5:1 contrast?1.4.3
SpeedCan players adjust the game speed?2.2.1 (Timing)
FontIs the font readable with distinct characters and adequate sizing?1.4.12
SoundCan users control audio volume independently?1.4.2
MotionAre there flashing elements (>3 per second)?2.3.1
InputIs the game fully playable with keyboard only?2.1.1
FeedbackAre game states (score, game over) clearly communicated?3.3.1

Suggested Timeline (50 min)

Fix 1: High-Contrast Colors & Score Display

Before (Inaccessible)

Grey background, small score text blending into the road, low-contrast car colors

After (Accessible)

Dark background, bold score on yellow rectangle, high-contrast car sprites

# Draw background, then score with high-contrast backing
DISPLAYSURF.blit(background, (0,0))
scores = font.render(str(SCORE), True, BLACK)
pygame.draw.rect(DISPLAYSURF, (255, 200, 0), (0, 0, 60, 65))  # Yellow box behind score
DISPLAYSURF.blit(scores, (15, 15))

Explain that the yellow rectangle behind the score creates a guaranteed high-contrast backdrop, regardless of the background image beneath it.

Fix 2: Adjustable Speed (Slow Mode)

# Setup
normal_mode = 60
slow_mode = 30
FPS = normal_mode

# In the event loop — toggle with spacebar
if event.type == KEYDOWN:
    if event.key == K_SPACE:
        if FPS == normal_mode:
            FPS = slow_mode
        else:
            FPS = normal_mode
        mode_message_time = time.time()

# Display current mode for 2 seconds after switching
if time.time() - mode_message_time < 2:
    mode_text = "SLOW MODE" if FPS == slow_mode else "NORMAL MODE"
    mode_msg = font_medium.render(mode_text, True, BLACK)
    rect = mode_msg.get_rect(topleft=(110, 20))
    pygame.draw.rect(DISPLAYSURF, (255, 200, 0), rect)
    DISPLAYSURF.blit(mode_msg, rect)
Discussion: Who Benefits from Slow Mode?
  • Players with motor disabilities who need more reaction time
  • Players with cognitive disabilities who process visual information more slowly
  • New players warming up before full speed
  • Young children or elderly players
  • Key insight: Accessible features often benefit everyone, not just disabled users

Fix 3: Accessible Font (Atkinson Hyperlegible)

# Use Atkinson Hyperlegible — designed for low vision readers
# Download from: https://brailleinstitute.org/freefont
special = "Atkinson-Hyperlegible-Regular-102.ttf"

font = pygame.font.SysFont(special, 60)
font_medium = pygame.font.SysFont(special, 40)
font_small = pygame.font.SysFont(special, 20)
game_over = font.render("Game Over", True, BLACK)

WCAG recommends fonts that are easy to read, have distinct characters (e.g., clear difference between I, l, 1), and generous spacing. Avoid pixelated, cursive, or thin-stroke fonts.

Fix 4: Volume Control

# Global variable
volume_level = 0.5

# In Player.move() — up/down arrows adjust volume
if pressed_keys[K_UP] and volume_level < 1.0:
    volume_level = volume_level + 0.1
    volume_level = min(volume_level, 1.0)
if pressed_keys[K_DOWN] and volume_level > 0.0:
    volume_level = volume_level - 0.1
    volume_level = max(volume_level, 0.0)

# On collision — play crash at user-set volume
crash = pygame.mixer.Sound('crash.wav')
crash.set_volume(volume_level)
crash.play()
Why Volume Control Matters

People with sensory sensitivities (common in autism) may find sudden loud sounds distressing. People who are hard of hearing may need higher volume. Giving users control over their audio environment is both an accessibility requirement and a UX best practice.

Extension Activities

For Advanced Students
  • Screen reader support: Add text-to-speech announcements for score changes and game over using pyttsx3
  • Color blind modes: Implement pattern/shape differentiation instead of relying solely on color
  • Remappable controls: Let players choose their own key bindings
  • Audit another game: Have students audit a commercial game or a classmate's project using the WCAG checklist
  • Research project: Investigate the Game Accessibility Guidelines (gameaccessibilityguidelines.com) and compare to WCAG
